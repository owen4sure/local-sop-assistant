# HEALTH_PANEL — 文件健康整理台

審核台的第四個分頁。**唯讀**（只標問題、不自動改文件）、需審核密碼。目的是隨著文件變多，協助人工發現「庫裡有沒有互相打架/重複/斷掉的文件」。

## 核心原則：flag-not-fix（只標記，不自動修）
- 系統**只負責「指出可疑點 + 給證據」**，所有修改都回到正常的「編輯 → 審核」治理流程，由人決定。
- 每個 flag 都帶 `type / severity / docs / evidence`；**沒有具體字串證據就不列出**（寧可漏、不要吵）。
- 每個 flag 可按「這不是問題」**消音**（dismiss）：把 flag 的簽章存進 `health_dismiss` 表，之後健檢不再列出。

---

## 判斷規則與門檻

### (a) 同名 / 近似重複  `type=duplicate` severity=high
兩份文件被標為重複，當：
- **標題正規化後相同**（去大小寫、空白、各種括號），或
- **去模板後內容 cosine ≥ 0.88**（`HEALTH_DUP_THRESHOLD`，沿用重複偵測的高把握門檻）

證據 = 兩份文件**最相似的 chunk 對**（`bestChunkPair`：對所有 chunk 兩兩算 cosine 取最高的一對）+ 各自的 owner / version。

> 為何 0.88：去模板後校準——真不相關 ~0.46–0.52、同領域不同子題 ~0.77、近雙胞胎 ~0.91。0.88 能抓到「幾乎重複」又不會把「同領域不同主題」誤判。

### (b) 引用失效 / 走鐘  `type=reference` severity=medium
掃每份文件的：
- frontmatter `similar_docs`
- 內文 `《...》` 參考（2–40 字）

若引用**對不到任何真實文件名** → 標記。判定「對得到」的規則（**只認準的方向**）：
```
resolves(ref) = norm(ref).length ≥ 3 AND 某個已知文件名包含 norm(ref)
```
- 只認「引用是某份真實文件名的片段」這個方向（`docName.includes(ref)`）。
- **不**用反向（`ref.includes(docName)`：引用文字吃掉某個短文件名）——那幾乎沒有合法案例，只會把斷掉的引用誤判成「有對到」而漏報。
- 最小長度 3（2 字中文片段太弱，容易巧合命中）。

證據 = 出現該引用的原始那一行。

### (c) 自相矛盾  `type=contradiction` severity=high（Phase 2，on-demand）
最謹慎的一項，**寧漏不誤**。分兩階段：

**候選（不靠 LLM、快）**：跨文件的 chunk 兩兩算 cosine，取每組文件對「相似度落在 `[0.80, 0.97)` 的最高 chunk 對」。
- 下界 0.80（`CONTRA_LOW`）：太不相似的根本不在講同件事。
- 上界 0.97（`CONTRA_HIGH`）：≥0.97 視為重複（交給 (a)），不在這裡。
- **排除中繼資料 chunk**（`isMetaChunk`：維護資訊/最後更新/版本/負責人/owner/version）→ 避免「各文件更新日期不同」之類假陽性。

**判定（LLM 稽核）**：每個候選對丟給對話 LLM，用嚴格 prompt 判斷「是否就**同一件事**給出互相衝突的指示/數字/條件/期限」：
- 必須能從 A、B **各引出一句互相衝突的原文**才算矛盾。
- 明確排除：語氣/用詞不同、各講各的、互補、不同流程/產品、各自的中繼資料不同。
- **雙重引文驗證**（`verifyQuote`）：LLM 引的兩句必須**一字不差**出現在原文（去空白比對、長度 ≥4），否則作廢。
- **自我複核**：再嚴格問一次，兩次都成立才收錄。

結果存 `health_contradictions` 表（每次掃描整批 replace）。`GET /sop-health` 把它併入清單（扣除已消音）。

> **掃描是背景執行**：按「🔍 掃描矛盾」後立刻回應，LLM 判斷在背景跑、前端輪詢進度（`scanned/total`）。不砍候選數（文件多了才用，覆蓋率要隨文件數成長）。代價：掃描期間會占用對話模型，問答會變慢——當維護動作離峰跑。

### (d) 結構缺口 / 內容空泛
這部分主要在**投稿預檢**（見 GOVERNANCE.md `contentCheck`）就攔下，不重複放進健康台：
- SOP 七段是否有空段（一句話總結 <10 字、適用對象 <4、標準步驟 <15、FAQ <8 → 標「偏空泛」）。
- 整體去模板字數 < 50、或同字大量重複（灌水）。

## 消音的簽章設計（避免誤殺）

| flag 類型 | 簽章 sig | 消音粒度 |
|---|---|---|
| duplicate | `dup:` + 兩文件名排序 | 該「文件對」 |
| reference | `ref:` + 文件名 + 引用字串 | 該「具體引用」 |
| contradiction | `con:` + 兩文件名排序 **+ 衝突引文的雜湊** | 該「具體矛盾」 |

> contradiction 的簽章**含衝突引文雜湊**（不只文件對）：對某個矛盾按「這不是問題」後，同一對文件日後出現「另一個」新矛盾**仍會被列出**；同一個矛盾重掃則維持消音。這修掉了「消音一次就把該文件對所有未來矛盾永久壓掉」的問題。

## API 一覽
| 端點 | 說明 |
|---|---|
| `GET /sop-health` | 唯讀回傳所有 flag（內網+密碼）；含 fileCount、dupThreshold、是否已掃過矛盾 |
| `POST /sop-health/dismiss {flagId}` | 消音一個 flag |
| `POST /sop-health/contradictions/scan` | 觸發背景矛盾掃描（立刻回 started+total） |
| `GET /sop-health/contradictions/status` | 輪詢掃描進度（running/scanned/total/found） |
| 健康台「🛠 編輯」 | 原地編輯（載入原文 → 改 → 持密碼直接更新）；載入失敗時禁止存檔，避免用空白覆蓋原檔 |
