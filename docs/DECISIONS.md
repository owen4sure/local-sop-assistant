# DECISIONS — 關鍵設計決策與理由

記錄「為什麼這樣做」，避免未來重建時重蹈覆轍。

---

## D1. 為何全本機、不接雲端 LLM
- **隱私 / 個資**：SOP 內含公司內部流程、聯絡人、個資。送到外部 LLM＝資料外流，不可接受。
- 因此**對話、embedding、reranker 三個模型全部跑在本機 127.0.0.1**，連對外網路都不是必要。
- 連結處理也保守：回答只能附「命中來源文件內」的 URL（白名單），不讓模型自由輸出外部連結。

## D2. 為何內網限定 + 寫入再加密碼（雙層）
- **第一層 `internalOnly`**：對外網域的請求一律擋掉寫入/審核 API（只認 LAN）。
- **第二層審核密碼**：即使在內網，會改變知識庫的操作（發布/刪除/直接更新/看草稿）都要密碼。
- 理由：內網不等於可信（同事誤操作、訪客裝置）；治理需要「誰、經過誰核可」。
- 密碼**絕不寫進原始碼**（env 或 gitignored 檔），否則一 commit 就外流。

## D3. 為何用 hybrid + reranker，而不是純向量
- 純向量分不開**雙胞胎文件**（兩份高度相似但屬不同產品/流程的 SOP）→ 會把 A 的答案配成 B。
- 加 **FTS5 關鍵字**補強：專有名詞、英文縮寫、低 cosine 但字面命中的 chunk。
- 加 **cross-encoder reranker**：把 `query` 與 `標題｜章節｜內容` 一起讀，精準分辨雙胞胎；分數也更適合當「有沒有命中」的訊號。
- **RRF 融合**（K=60）而非分數相加：兩路用「名次」融合，免去 cosine 與 bm25 量綱不同的問題。

## D4. 雙訊號門檻怎麼校準（reranker 主、cosine 副）
- 觀察到的分數分佈：
  - reranker 真陽性 ≥ 0.19；沾邊離題雜訊（只是某字重疊）~0.02 → 設 **RERANK_THRESHOLD = 0.1** 擋雜訊又不傷真陽性。（原本 0.01 太鬆，會把不相關文件灌進 context。）
  - 「換句話問」的極端案例 reranker 會整題偏低，但 cosine 仍 ~0.617 → 設 **COSINE_GATE = 0.60** 當「整題最後手段」救它；離題 cosine ≤ 0.574 不會誤救。
- 關鍵：**cosine 是「整題的 fallback」，不是逐 chunk 的 OR**。只有 reranker 對整題都沒信心時才看 cosine，否則一律以 reranker 為準 → 同時壓住誤報與漏報。
- ⚠️ **換 embedding 或 reranker 模型，這兩個門檻要重新校準**（不同模型分數分佈不同）。校準方法見 D8。

## D5. 注入策略＝命中 chunk，不是整份文件；MAX_CONTEXT_CHARS 的教訓
- 餵整份文件會稀釋重點、拖慢生成。改餵「命中的 chunk」並標上 `文件名｜章節`。
- **踩雷**：context 上限原設 900 字，會把 reranker 已認可的答案 chunk 硬砍掉 → 「正確文件命中卻沒被餵進去 → 答查無」。
- 修正：提到 **2000**——讓門檻認可的 chunk（受 TOP_N_CHUNKS=5 限制）都餵得進去。原則：**尊重 reranker 的精準判斷**，不要用字數上限去否決它。

## D6. 為何沒用 MMR（最大邊際相關）
- MMR 用來「去除餵進去的 chunk 彼此太相似」以增加多樣性。
- 本系統的多樣性已由別的機制處理：FTS 的 **per-doc cap = 3**（單一文件不灌爆候選）、來源去重（sources 每份文件只列一次）、reranker 精排。
- 對「單一主題的 SOP 問答」來說，多樣性不是痛點，**精準才是**；MMR 反而可能把最相關的兩段拆掉。故**評估後不採用**，保持管線簡單。

## D7. 查詢正規化：空格 == 標點
- 使用者不會「正確使用頓號」。把所有軟分隔（半/全形空格、頓號、逗號、斜線、分號、冒號…）統一成單一半形空格 → 「A、B、C」與「A B C」對向量/FTS/reranker 三路都變成相同輸入。
- FTS 多詞用 **OR 不是 AND**：多產品/多主題問題不會因「要同時包含所有詞」被縮窄到 0 命中。
- docFreq 過濾：命中 >60% 文件的通用詞/品牌詞（會灌爆候選池、把其他文件擠掉）丟棄。

## D8. 門檻校準方法（給重建者）
1. 準備一份評測集：每題標「應命中哪份文件」+ 一批「應查無資料」的離題問題。
2. 跑檢索（只看檢索，不接 LLM），印出每題的 topRerank / topCos / gateVia。
3. 看真陽性的 rerank 分佈下緣、離題的上緣 → 取中間當 `RERANK_THRESHOLD`。
4. 對 reranker 救不到但語意對的「換句話問」案例，看其 cosine → 定 `COSINE_GATE`。
5. 目標：真陽性全進、離題全擋（查無資料準）。

## D9. 治理：用標題當文件身分的取捨與補丁
- 簡單起見，文件以**標題/檔名字串**為身分（沒有不可變 ID）。代價：撞名會覆蓋、改名會生孤兒。
- 沒有改成「全域唯一 ID」（會牽動索引、git 檔名、review UI），而是**在核准與預檢加防呆**：撞名→要確認覆蓋、改名找不到原檔→要確認新建、刪不存在→404。
- 這是「在現有抽象上補防呆」與「重構成 ID 制」之間的務實取捨；若文件量再大、改名頻繁，建議升級成 ID 制。

## D10. 冷啟動防呆（本機推論引擎會踢模型）
- 本機引擎記憶體有限，載入別的模型會把對話模型踢出 → 冷載入第一發可能回空字串。
- 對策：空白自動重試、啟動預熱、定期保溫（每 4 分送極短請求）、真的失敗回明確重試訊息（不把空答案配來源送出）。

## D11. 增量索引 + chokidar 雙保險
- 用「內容簽章 + 模型名」判斷是否重算 embedding，沒變就跳過（省算力）。
- 發布時主動 reindex + chokidar 監看檔案變更也 reindex（冪等，雙保險）。
- 換 embedding 模型 → 啟動時自動清掉舊維度向量並重建。

## D12. 矛盾偵測：寧漏不誤 + 雙引文驗證 + 背景掃描
- 自相矛盾是最容易誤報的（同領域文件本來就像）。設計上**極度保守**：相似度帶限制 + 排除中繼資料 + LLM 嚴格 prompt + **引文必須一字不差出現在原文** + 自我複核兩次。
- 寧可漏掉一些，也不要一堆假矛盾把人淹沒（否則沒人會看）。
- 掃描慢（每候選對要 LLM 判斷）→ 改**背景執行 + 進度輪詢**，不卡住請求；且**不設候選上限**（這功能本來就是文件多了才用，覆蓋率要隨文件成長）。

## D13. system prompt 固定不變（省 prefill）
- system prompt 不含 {context} → 推論引擎可快取 system 前綴。參考資料放 user 訊息。
- 省下每次請求的 system prefill，對本機有限算力的延遲幫助明顯。
